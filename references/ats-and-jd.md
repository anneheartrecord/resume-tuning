# ATS 与 JD 匹配标准

> 用途：让简历既「过机器关」（ATS 可解析），又「贴目标岗位」（命中 JD 关键词）。配套 skill：`resume-tuning`，机械检查脚本：`scripts/ats_check.py`。
> 与 `resume-standards.md`（价值观/底线）、`resume-writing.md`（字数/强调硬指标）平级。

---

## 〇、第一约束：surface 真实，绝不堆砌（anti-stuffing）

JD 匹配是**让你真有的亮点用招聘方的语言被看见**，不是**伪造你没有的**。

- 堆砌虚假关键词能骗过 ATS，但 ATS 之后是人眼和背调——面试官一问「你这个 Kubernetes 项目具体做了什么」答不上来，当场社死，比没写还糟。
- 这是 resume-tuning 区别于「关键词填充」类工具的根本立场，和 `[DATA NEEDED]`「不替你编数字」是同一条护城河的延伸。

**所以缺失关键词不是「往简历里塞词」的信号，而是先分三档，只有第一档能改简历。**

---

## 一、ATS 是什么 /「ATS 友好」到底指什么

ATS（Applicant Tracking System，申请人追踪系统）是企业批量**解析、存储、检索、初筛**简历的软件。多数中大型公司先过 ATS 再到人眼，简历若解析失败或检索不到，根本到不了招聘官面前。

「ATS 友好」= 机器能正确解析 + 在关键词检索里被命中。六条具体要求：

| # | 要求 | 为什么 | resume-tuning 现状 |
|---|---|---|---|
| 1 | **文本层可提取**（非图片/扫描件） | 图片简历 ATS 读出来是空 | ✅ WeasyPrint 输出真实文本 PDF，天然满足（仍要核验，见 Step 1 扫描件路径） |
| 2 | **单栏、线性阅读顺序** | 多栏/浮动框会让解析错位、串行 | ✅ 三套模板都是单栏 flex 纵向流，无多栏分割 |
| 3 | **标准章节标题** | ATS 靠标题切分「经历/教育/技能」 | 需用机器认得的词，见第三节对照表 |
| 4 | **联系方式可解析** | email/电话要是纯文本，不嵌图标字体 | 用 `<a href="mailto:">` + 纯文本电话 |
| 5 | **关键词覆盖** | ATS 按 JD 关键词检索/打分 | 本次新增能力，见第二节 |
| 6 | **关键信息不只放进表格/图片/文本框** | 这些区域常被解析丢弃 | ✅ 公司名/职位/技能都是正文文本 |

> 不必为 ATS 牺牲排版美感：resume-tuning 的 HTML→PDF 单栏文本路线，第 1/2/6 条本就达标。真正要主动做的是第 3/4/5 条——标准标题、可解析联系方式、关键词覆盖。

## 二、缺失关键词三档处理（核心流程）

agent 从 JD 抽出关键技能/工具/职责/资历词，与用户**真实经历**比对，分三档。`ats_check.py` 只做诚实计数（覆盖了哪些、缺哪些），**分档判断由 agent + 用户做**。

| 档 | 含义 | 动作 | 例子（JD 要 "Kubernetes"） |
|---|---|---|---|
| ✅ 已覆盖 | 简历已明确写到 | 不动 | 简历已有「用 K8s 编排 200+ 实例」→ 保持 |
| 🟡 有但没突出 | 用户**真有**，但措辞没用 JD 的词 / 埋太深 | **改写 surface**：用 JD 术语重述、提到显眼位置（自我评价首条 / 经历首句 / 技能栏首行） | 简历写「容器编排」却没出现 "Kubernetes" → 改成「Kubernetes 容器编排」并前置 |
| 🔴 真没有 | 用户实际不具备 | **绝不编**。列为诚实技能缺口反馈用户，可追问是否有相关经历被漏写，但不写进简历 | 用户从没碰过 K8s → 不写。告知「JD 要求 K8s，你简历无相关经历，建议补真实经历或如实留空」 |

判定 🟡 还是 🔴 的唯一标准：**用户是否真有这段经历**。拿不准就追问用户，不替他归档。

## 三、标准章节标题对照（中英文）

ATS 靠标题识别区块。用下表里「推荐」的标题，别用花哨自造词（如「我的旅程」「能力雷达」），机器认不得。

| 区块 | 推荐英文标题 | 推荐中文标题 | 避免 |
|---|---|---|---|
| 工作/项目经历 | `Experience` / `Work Experience` / `Employment` / `Projects` | `工作经历` / `项目经历` / `工作经验` / `项目经验` / `实习经历` | 「职业旅程」「我做过的事」 |
| 教育 | `Education` | `教育经历` / `教育背景` | 「求学历程」 |
| 技能 | `Skills` / `Technical Skills` | `技能` / `专业技能` / `技术栈` | 「能力雷达」「技能树」 |
| 自我评价（可选，ATS 非必需；`ats_check.py` **不对该档做识别**，不参与硬伤判定） | `Summary` / `Profile` | `自我评价` / `个人简介` | 「关于我」可用但优先标准词 |

> 本表的 experience / education / skills 三档就是 `ats_check.py` 的 `SECTION_PATTERNS` 识别词典（Summary 不识别）。新增同义标题时，脚本与本表同步更新，两边别漂移。

## 四、ATS 自检清单（定稿前逐条扫）

定稿 PDF 渲染后、交付前跑 `python3 scripts/ats_check.py <定稿.pdf> --jd jd.txt`，对照下面逐条过：

1. **文本层可提取**：脚本能提出正文（不是图片型 PDF）。提不出 → 检查渲染或扫描件路径。
2. **标准章节可识别**：Experience / Education / Skills（或中文对应）**至少认出 2 个**（Skills 对部分岗位非必需，故门槛是 2/3 而非全部）。认出不足 2 个 → 改标题用标准词。
3. **联系方式可解析**：至少 email 能被正则抽到（电话可选）。抽不到 → 确认是纯文本而非图片/特殊字体。
4. **一页 + 无占位符泄漏**：复用渲染期检查，`{{...}}` / `[DATA NEEDED]` / 字体占位符不得残留。
5. **关键词覆盖率达标**：给了 JD 时看覆盖率 `M/N` 和缺失列表。缺失项**先分三档**（第二节），只补 🟡 档，🔴 档如实反馈。
6. **阅读顺序正常**：姓名/抬头出现在提取文本最前部（防多栏/页眉错位，尤其 `modern` 彩色页眉版式）。

可读性硬伤（1/2/3/6）未过 → 回 Step 4/6 修，重渲染。覆盖率（5）**默认是参考不是硬门槛**；只有显式给 `ats_check.py --min-coverage <RATE>`（0-1）时，覆盖率低于该值才会让脚本非零退出。按三档处理缺失项，别为提分堆词。

## 五、关键词覆盖率怎么读（理解脚本输出）

`ats_check.py` 输出的覆盖率 `M/N` 是**诚实计数**，不是黑盒匹配分：JD 给了 N 个关键词，简历命中 M 个。

匹配语义（写进脚本，agent 要理解）：

- **英文关键词按 token 边界匹配**：`java` 不命中 `javascript`、`c` 不命中 `c++`/`c#`（边界字符含 `+ #`，避免短关键词被符号延伸的 token 吃掉、抬高覆盖率）。`C++`、`Node.js`、`.NET` 这类带符号词按转义处理；句末句号（`Python.`）仍算命中。**连字符归属本就歧义**（`go`↔`go-lang`、`react`↔`react-native`），脚本不强判，留给 agent 抽词消歧。
- **含中文的关键词按子串匹配**：中文无词边界，`分布式存储` 在「构建分布式存储系统」里算命中。
- **可选模糊匹配**（装了 `rapidfuzz` 且跑时加 `--fuzzy 90` 才启用；不加 `--fuzzy` 即使装了也只走精确）：只认**前缀式词形变体**（`Postgres`↔`PostgreSQL`、`nodejs`↔`node`），且**短词（<5 字）不参与**——挡掉 `sql`↔`sqs`、`cs`↔`css` 这类短词假阳。互不为前缀的近似异名（`react`↔`preact` 相似度高达 90.9 却是两个框架）会被拒。模糊命中**单独标注、待 agent 复核**，不与精确命中混同。**注意**：模糊解决不了缩写/同义（`K8s`↔`Kubernetes` 是别名不是词形变体）——这类靠 agent 抽词时**同时给出别名**（把 `Kubernetes` 和 `K8s` 都作为关键词传入），脚本不臆测同义词。

读覆盖率的正确姿势：低覆盖率先逐条看缺失项是 🟡（可补，改写 surface）还是 🔴（真没有，如实留），**不是盲目把缺失词塞进简历**。
