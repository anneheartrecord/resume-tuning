# 简历调整标准

> 用途：诊断和优化简历时的判断标准。配套 skill：`resume-tuning`。

---

## 〇、输出底线（硬性，无论原始简历多普通都必须达到）

生成方不是誊抄机，是升级器。哪怕来源简历平庸、只是功能罗列，产出也必须满足三条：

1. **尽可能突出亮点**：主动从平淡内容里挖出最有价值的点（最难的技术、最像主导的事、最有数字的结果、最稀缺的经历），放到最显眼处——自我评价第一条、经历第一段、要点首句。没人帮他提炼，生成方来。
2. **保持良好结构**：清晰分块、按身份/影响力排序、一行一句、最亮点前置，扫读 10 秒能抓到重点。
3. **每条经历遵循 STAR**：背景 / 你负责什么 / 你做了什么 / 量化结果。来源只有「做了 X」的功能描述，要补出 Action→Result 的因果；结果数字缺失标 `[DATA NEEDED]` 追问，但 STAR 骨架先搭好。

---

## 一、简历的本质（先立心智模型）

简历不是个人信息登记表，是一份**销售文案**。面试官平均只花 **10 秒**决定你值不值得聊。所有技巧都服务于一个目标：10 秒内让他得出「这个人好像很厉害，我想面面」。

一句话标准：**好简历是一个钩子矩阵，每一行都在埋一个你最擅长回答的问题，引导面试官走你的剧本。**

---

## 二、好简历的标准（9 条核心）

### 1. 只能有一页
没有商量余地。超过一页，面试官大概率不翻第二页。一页的限制会逼你**筛选**——砍掉可有可无的，只留最能证明价值的。放不下，说明还没想清楚自己最值钱的是什么。

### 2. 一行一句话，一句话一行
每个点尽量一句话、不换行。面试官跳着看，要点写三行他只看第一行前半段就跳过。也强迫你精简表达。

### 3. 最亮的点放最前面
默认时间倒序，但**有特别亮眼的经历就打破时间顺序提到最前**。前 5 秒注意力最集中，最强的武器第一时间亮出来。规则是死的，说服力是活的。
- **最强优势决定排序**：先确认求职者最想突出什么（如「能独立从 0 到 1」「全栈 / Founder 心态」），就把它放自我评价第一条，而不是默认按技术能力排。同一类亮点（如多个「从 0 到 1 主导」的项目）让它们形成 pattern，互相强化。

### 4. 量化一切，遵循 STAR
每段经历用 STAR 组织：Situation 背景 / Task 你负责什么 / Action 你做了什么 / Result 结果（用数字）。
- 坏：负责服务端性能优化，提升了系统性能。
- 好：主导核心接口优化，P99 从 800ms 降至 120ms，QPS 提升 5 倍，支撑 10 万级并发。
- **没有数字的经历，在面试官眼里就是空气。** 每段 bullet 控制在 3–5 个，最多不超过 10 个。

### 5. 不要堆技能列表
裸列 20 个技术名词＝一堆没上下文的关键词，面试官的反应是「所以呢？」。**让项目经历证明技能**——写了高并发系统用 Redis 缓存、消息队列解耦，面试官自然知道你会，而且知道你真用过、解决过问题。技能栏可保留作补充，别指望它加分。
- 技能栏**把目标方向放第一条**（主攻某方向就把它放技能栈最前），别让旧主线压住新方向。

### 6. 不要写空话，要有论据（最高频的坑）
「自驱力强 / 学习能力强 / 责任心强」每份简历都写，写了等于没写。空洞形容词零信息量。**一个具体事实胜过一百个形容词。**
- 自驱力强 → 半年减重 50 斤 / 非科班自学拿到大厂 offer / 技术博客累计阅读 15 万+。
- 想突出某项软实力，就用一个具体的、可量化的事实顶住它。**产出型证据（开源项目、上线数据、GitHub 链接）一定要点名**，「我看过源码」远不如「我看完还开源了一个项目」。

### 7. 埋钩子
不要把一件事说太细，**留白让面试官好奇**。「主导 XX 平台架构，上线一周用户破 1000+」——他自然想问怎么做到的，而这恰恰是你最擅长答的。每行埋一个你准备好深度回答的问题。

### 8. 实事求是
不要把参与写成主导、了解写成精通、别人做的写成自己做的。简历能骗过筛选，面试一问就穿帮（问你为什么选 A 不选 B 答不上来＝当场社死）。
- **「参与」和「主导」的措辞必须精确**（只参与的就写参与，别为了好看都写主导）。
- **放大的数字要能对答**，背调 / 追问要对得上。
- well done 和 bad done 都讲，反而体现复盘意识。

### 9. 个人总结（自我评价）怎么写——很重要，推荐写
只写三句话，对应**过去 / 现在 / 未来**：
1. 你在行业 / 专业取得的结果（用事实建立第一印象，如「大厂 T6 / 开源 500+ star」）。
2. 你是一个什么样的人，**带论据**。
3. 你想成为什么样的人（方向感）。
- **三条格式要对齐**：每条都用「**标签**：内容」的形式，别一两条有标签、一条没有。
- **「未来」那句别喊口号**（「立志成为顶尖 X」很空），更好的做法是把方向感埋进抬头的一句话定位里，用经历顺序自然讲出「我往哪走」。
- **避免减分词**：简历上别自我打折，「意识较强」不如「意识强」。

## 三、坏简历的典型问题（诊断时逐条扫）

| 问题 | 表现 | 怎么改 |
|---|---|---|
| 当成信息登记表 | 教育 / 经历 / 技能从上到下平铺，没有重点 | 按销售文案重排，最亮点前置 |
| 超过一页 | 内容堆砌、舍不得砍 | 压缩弱经历给强经历让篇幅（如把一段几个月、与目标方向无关的实习压成 1 行） |
| 没有量化 | 「提升了性能」「负责了 XX」 | 补数字，套 STAR |
| 空洞形容词 | 自驱力强 / 学习能力强 / 责任心强 | 每句自我评价都问「我有论据吗」，没有就删或补 |
| 裸技能列表 | 一排技术名词，不知精通还是了解 | 用项目证明；技能栏只作补充 |
| 夸大 | 参与写成主导、了解写成精通 | 措辞精确，实事求是 |
| 时间线过期 | 已离职公司还写「至今」、职级 / 年龄过期 | 入职新公司后立刻更新倒序时间线和在职状态 |
| 标签 / 格式不齐 | 自我评价有的带标签有的不带、数字一会儿汉字一会儿阿拉伯 | 统一格式，**数字统一用阿拉伯数字** |
| 缺产出钩子 | 只写「系统解析过 X 源码」却不提开源项目 | 点名开源作品 + GitHub 链接（工程师岗 GitHub 接近刚需，方便面试官一键看代码） |

## 四、错别字 / 一致性校对清单（每次必扫）

反复出现的低级错误，定稿前逐项过一遍：

1. **专有名词大小写**：`DevOps`（不是 Devops / devops）、`Kubernetes`、`GitLab`、`PostgreSQL`、`GitHub` 等，全文统一。
2. **分隔符**：并列技术名用顿号 `、` 或斜杠 `/`，**不要用反斜杠 `\`**（`A\B\C` 是错的，应为 `A、B、C`）。
3. **多字 / 漏字**：改写残留（如句中多出或缺失一个字）、句尾标点缺失。逐句通读一遍能抓到。
4. **同一份内两种写法**：同一个词在一处写一种、另一处写另一种——必须统一。
5. **数字格式统一**：要么全阿拉伯数字，要么全汉字，别混用（建议技术简历统一用阿拉伯数字，与 1000+ / 86.2% 这类指标对齐）。
6. **职位名 / 抬头**：职位名拼写、是否与背调一致；年龄信息是否过期（可直接删年龄，多数地区非必填）。

## 五、给反馈的方式（流程约定）

- **默认用文本方式反馈，不擅自动手改**（尤其简历常是 PDF，改不了源——错别字要让用户回源文件改完重新导出）。
- 反馈分三档，让用户好决策：**必须改的错别字 / 硬伤** → **强烈建议改的内容点（带论据）** → **可选的锦上添花**。
- 每条建议都给**理由和论据**，不空喊「这样更好」。
- 归档 / 移动文件时：先确认真实目录，**不替换 / 不覆盖其他简历**，除非用户明确说「替换掉某份」。

## 六、上线前 anti-patterns 清单（定稿前逐条扫，精简自 Kami）

定稿一份简历前，对照下面逐条过。每条「坏 → 改」。

**内容空洞**
- 形容词堆砌无数字：「在各项指标上取得显著增长」→ 给数字「营收同比增长 34% 到 1200 万」。
- 开头套话：「在快速变化的时代…」→ 删掉，直接从第一个真实主张开始。
- 把标题换句话重说一遍：标题「营收增长」+ 正文「我们的营收增长很可观」→ 正文必须带标题没有的信息。
- 模糊时间：「最近上线」「不久后」→ 钉到具体季度「2026 Q1 上线」。

**数字造假**
- 假精度：「市场份额 23.7%」却没来源 → 要么给来源，要么写「约 24%」。
- 圆整数装精确：源数据是「约 1 万」却写「正好 10000」→ 跟源数据精度一致。
- 无基准的对比：「比同类快 3 倍」没说跟谁比 → 点名对比对象和基准方法，否则删。

**结构套路**
- bullet 无结果：「负责一个跨职能团队」→ 动作+规模+结果「带 8 人团队交付 v2.0，流失率降 15%」。
- 技能栏填充词：「沟通、团队协作、解决问题」→ 写具体技能 + 用在哪，否则砍掉这栏。
- 夸大：参与写成主导、了解写成精通 → 措辞精确，背调对得上。

**视觉过度**
- 一页超过 3 处强调色 → 一种强调色，层级靠字重和字号，不靠多种颜色。
- 满屏高亮 `.hl` → 每 80–150 字最多一处，只标最重要的数字或名词（见 `resume-writing.md`）。
