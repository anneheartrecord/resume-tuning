# 技术岗简历调整流程（resume-tuning）

## Overview

帮**技术人员**诊断、优化、校对简历。这个 skill 只是流程，**判断标准在 `references/resume-standards.md`**：先把那份「好简历标准 + 坏简历问题 + 校对清单」读进上下文，再按下面步骤执行。

只面向技术岗，所以默认带这些技术简历的偏好：GitHub / 开源作品接近刚需、技能用项目经历证明、技术专有名词大小写要规范（DevOps / Kubernetes / GitLab）、量化用技术指标（P99、QPS、节省成本、实例数、覆盖节点数）。

## 使用条件

触发场景：
- 「看下我的简历，有哪些可以优化」
- 「帮我把 XX 这段经历加进简历 / 调整简历 / 优化一下」
- 「这版简历还有什么问题 / 帮我校对一下错别字」

不适用：
- 非技术岗简历（产品、运营、设计等）——本 skill 的技术偏好不一定合适，先和用户确认。
- 写求职长文 / 面经文章——那是写作任务，不走这里。

## 处理流程

### Step 0：读标准
打开 `references/resume-standards.md`，把规则吃进上下文。本 skill 只管流程，标准在那里。

### Step 1：拿到简历本体
- 用户给了文件（PDF / Markdown / 图片）就直接读。
- 没给就问用户要文件或路径。
- 注意区分「写简历的方法论文档」和「简历本体」——别把指导原则当成简历来改。

### Step 2：摸清结构
把简历拆成模块：抬头 / 联系方式、自我评价、工作经历（倒序）、专业技能、教育经历。先问清楚或从对话里确认用户**最想突出的优势**，它决定排序。

### Step 3：错别字 + 一致性校对（必扫）
按标准文档的校对清单逐项过：
1. 专有名词大小写统一（DevOps / Kubernetes / GitLab / PostgreSQL 等）。
2. 并列项分隔符用顿号或斜杠，不要用反斜杠 `\`。
3. 多字 / 漏字、句尾标点缺失——逐句通读能抓到。
4. 同一份内同一个词出现两种写法。
5. 数字格式统一（全阿拉伯数字，避免「7 倍 / 七倍」混用）。
6. 职位名拼写；年龄等过期信息。

### Step 4：结构与内容优化（按好简历标准逐条）
- 一页？最亮点是否在第一屏？
- 时间线最新？已离职的公司别写「至今」。
- 每段经历有数字（STAR）？没有就标出来让用户补。
- 自我评价：三条「标签：内容」对齐？每条带论据？最强优势排第一？「未来」别喊口号？避免「较强」这类减分词？
- 技能栈：目标方向放第一？是否用项目证明而非裸列？
- **产出型钩子**有没有点名：开源作品、GitHub 链接、上线数据。
- 措辞精确：参与 vs 主导别夸大；放大的数字能不能对答（实事求是）。
- 弱经历是否该压缩给强经历让篇幅。

### Step 5：给反馈（默认文本，分三档）
**默认只诊断、用文本说，不擅自动手改**——简历多是 PDF，改不了源，错别字要让用户回源文件改完重新导出。
输出分三档让用户好决策：
1. **必须改**：错别字、硬伤、过期时间线。
2. **强烈建议改**：缺论据、缺产出钩子、排序、夸大——每条给理由。
3. **可选**：锦上添花。
最后给一句整体判断（这版相比上版进步在哪、核心还差什么）。

## 注意
- 这是诊断 + 建议型 skill，主导权在用户：他说「应该没问题了」不代表真没问题，发现残留错别字要如实指出。
- 归档 / 移动文件前先确认真实目录，**不覆盖其他简历**，除非用户明确说「替换掉某一份」。
